# Preview-окружения

> Статус: проектная заметка, не реализовано. Описание того, как я бы добавил per-PR preview-окружения поверх существующей основы, что это стоит и почему пока не стоит строить. Для одного сервиса и маленькой команды общий namespace `dev` плюс `make drill` покрывают большую часть ценности. Preview'ы начинают окупаться когда несколько человек одновременно катят в один сервис и мешают друг другу в `dev`.

## Идея

Каждый открытый PR получает короткоживущий изолированный деплой. Ревьюеры кликают по реально работающему инстансу вместо того чтобы воображать дифф. При закрытии PR всё сносится.

Чарту не нужны новые шаблоны — preview'ы это просто ещё один набор Helm-values, направленный в per-PR namespace.

## Как ложится на то, что есть

Чарт уже достаточно параметризован без изменений:

- **Namespace на PR** — `orders-api-pr-<number>`. Создаётся preview-джобой, размечается так же как Terraform размечает dev/stage, поэтому существующие NetworkPolicy и Kyverno-селекторы `orders-api-*` применяются без изменений.
- **Имя релиза на PR** — `helm upgrade --install orders-api-pr-<number>`.
- **Образ** — иммутабельный `sha-<gitsha>`, уже собранный CI для этого коммита. Никакой особой сборки; preview переиспользует тот же артефакт что и продвижение.
- **Values** — стартуем от `values-dev.yaml` (1 реплика, дёшевые ресурсы, `debugEndpoints: true`, HPA + PDB выключены) с маленьким оверлеем:
  - `ingress.host: pr-<number>.preview.example.com`
  - жёсткие потолки ресурсов
  - `autoscaling.enabled: false`, `pdb.enabled: false`

## Жизненный цикл

```
PR открыт / запушен  ->  собрать sha-тег (существующий CI)  ->  helm upgrade --install
                         в orders-api-pr-<n>  ->  оставить URL комментарием в PR
PR закрыт / смерджен ->  helm uninstall + удалить namespace
Ночная уборка        ->  GC любого preview-namespace, чей PR больше не открыт
                         (подстраховка на случай упавшей cleanup-джобы)
```

Ночная уборка не опциональна. Осиротевшие namespace'ы, тихо жгущие квоту — самый частый способ протухания preview-сетапов. Нужно сверяться со списком открытых PR и удалять всё устаревшее.

## Стоимость и изоляция

**Стоимость.** Каждый preview — реальные CPU и память. Держать под контролем: 1 реплика, низкие requests, жёсткий ResourceQuota и TTL чтобы ничего не пережило свой PR. На kind почти бесплатно. На реальном кластере — главная причина держать preview'ы маленькими и агрессивно GC'шить.

**Изоляция.** Только на уровне namespace, чего достаточно для stateless-сервиса. Как только orders-api понадобится база данных — preview означает ещё и провижининг одноразового хранилища или сидинг общего с per-PR схемой. Вот где настоящая сложность, и настоящая причина отложить — не вычисления.

**Секреты.** Preview'ы должны использовать только непродакшен-креды, подключаемые из того же пути External Secrets / SOPS что и dev. Никогда stage.

**DNS/ingress.** Wildcard `*.preview.example.com`, направленный на ingress-контроллер, делает per-PR хостнеймы беспроблемными.

## Почему отложено

Один сервис, маленькая команда — конкуренция в `dev` низкая. Выигрыш небольшой.

Базы данных пока нет. Сложная часть preview-окружений (эфемерные данные) ещё не существует. Строить лёгкую часть сейчас — преждевременно.

`make drill` уже позволяет ревьюеру локально прогнать режимы отказа, а это большая часть того, для чего preview использовался бы на этом этапе.

Когда команда вырастет или появится второй сервис, это добавляется за полдня: оверлей `preview.yaml` и одна GitHub Actions-джоба по событиям `pull_request` `opened/synchronize/closed`, переиспользующая уже имеющуюся обвязку деплоя.
